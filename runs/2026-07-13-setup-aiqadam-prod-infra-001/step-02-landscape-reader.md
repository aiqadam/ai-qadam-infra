## Summary
`pro-data-tech-prod` (95.46.211.224, 16 vCPU / 31 GiB RAM / 339 GB disk, Docker CE 29.6.1, security baseline fully hardened) currently runs exactly one workload: a healthy 7-container Penpot 2.16 Docker Compose stack (project `penpot`, dir `/opt/penpot/`) with its frontend bound to `0.0.0.0:9001`, fronted by an nginx vhost (`/etc/nginx/sites-available/penpot.aiqadam.org`) and a Let's Encrypt cert expiring 2026-10-09; UFW already allows 22/80/443/tcp and no other app checkout or Compose project exists on the host yet, so port 9001 is the only occupied slot to avoid. The `aiqadam.org` Cloudflare zone is shared, heavily-used, third-party infrastructure (33 records total, only 2 owned by this repo — `penpot.aiqadam.org` and `qa-uz.aiqadam.org`); critically, the **apex `aiqadam.org` A record currently exists and points to `212.20.151.29`**, an undocumented third-party Coolify-like PaaS host in Uzbekistan (Globe Cloud LLC) that also serves the wildcard `*.aiqadam.org` and the mail platform — it does **not** point at `pro-data-tech-prod` today, so T-0111's Cloudflare step is a **repoint of an existing shared-zone record**, not a fresh greenfield create (unlike T-0110's `qa`/`qa-uz` case, which was a brand-new name). The app-registry's AiQadam QA section is fully populated (T-0110, done); the prod section does not exist yet and must be created from scratch by this task's landscape-updater step. Finally, and most importantly for the designer: T-0110 discovered that `apps/api`'s tenant-resolution middleware only treats a hostname's **leftmost DNS label** as a tenant-code candidate when that label is **exactly 2 characters**; anything else (any length) falls through to `DEFAULT_TENANT_CODE='uz'`. For a bare apex domain `aiqadam.org` with no subdomain, the "leftmost label" as parsed by a `hostname.split('.')[0]`-style tenant middleware would be `aiqadam` itself (7 characters) — not a subdomain in the qa-uz sense at all. This is very likely to behave the same way `qa-uz.aiqadam.org` did (7-char label fails the 2-char check → falls through to the `uz` default and returns 200), but this has **not been confirmed against source** for the apex case specifically, and the middleware's actual host-header parsing logic (e.g., whether it strips a port, whether `req.hostname` vs `req.headers.host` is used, whether there's an apex-specific special case) is unverified. This must be flagged as an explicit open risk, not assumed safe by analogy.

### Tenant-routing / apex-domain risk (flagged per step-01 instruction)

T-0110's app-level root-cause finding (quoted from the task file): the app's tenant-resolution middleware (`apps/api/src/modules/tenants/tenant.middleware.ts`) "treats any 2-character leftmost hostname label as a literal tenant code," and a hostname whose leftmost label is **not exactly 2 characters** falls through to `DEFAULT_TENANT_CODE='uz'`. `qa` (2 chars) hit the literal-tenant-code branch and 400'd because `qa` isn't registered; `qa-uz` (5 chars) missed that branch and fell through to the default successfully.

For the bare apex domain `aiqadam.org` (no subdomain at all), there is no "next label down" in the conventional subdomain sense — depending on how the middleware actually derives its "leftmost label" (e.g., `req.hostname.split('.')[0]`), the apex case would most likely parse as `aiqadam` (7 characters, the second-level domain itself) — not a 2-character string, and by the same logic documented for `qa-uz`, this would also fail the literal-tenant-code check and fall through to `DEFAULT_TENANT_CODE='uz'`, returning 200 the same way `qa-uz.aiqadam.org` does today.

**However, this is an inference by analogy, not a confirmed fact** — none of the landscape files, nor the T-0110 task file's quoted root-cause text, describe what the middleware does when the Host header has no subdomain at all (i.e., a 2-label FQDN rather than a 3-label one). Specific unknowns:
- Whether the middleware's label-splitting logic special-cases a bare/apex Host header differently (e.g., treats a 2-label hostname as "no tenant label present" and takes some other branch entirely, possibly one that isn't the same default-fallback path).
- Whether `www.aiqadam.org` (if the user opts into it per the open `www` question) would parse differently again — `www` is 3 characters, so by the documented 2-char-check logic it would also miss the literal-tenant-code branch and fall through to default, but again unconfirmed against source.
- Whether any NON_TENANT_LABELS allowlist (referenced obliquely in the app-registry's "Tenant-resolution nuance" note as a thing that could "silently change this hostname's behavior") already contains special handling for bare apex or `www` that differs from the generic length-check path.

This is exactly the kind of app-behavior fact that cannot be safely assumed from infrastructure landscape alone — the QA precedent was only discovered by reading the actual `tenant.middleware.ts` source during execution, not by design-time inference. **Recommend the solution-designer either (a) treat this as a source-verification pre-step within the plan (read `apps/api/src/modules/tenants/tenant.middleware.ts` from the app's git checkout before finalizing the nginx/DNS plan), or (b) flag it explicitly as a NEEDS_APPROVAL open question for the user, since a 400 on the production apex domain — unlike QA's health-check-only slice — would be a directly user-visible production failure.**
