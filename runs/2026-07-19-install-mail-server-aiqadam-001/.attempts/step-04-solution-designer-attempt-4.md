### The core design decision this retry must resolve: `Bootstrap.generateDkimKeys` and `Bootstrap.requestTlsCertificate`

**Resolved value: both `false`.** Reasoning, stated explicitly per the retry brief's instruction not to assume this silently:

- **`generateDkimKeys: false`.** The live `Bootstrap` schema shows this defaults to `true` and, per its description, would have Stalwart auto-generate DKIM signing material for `defaultDomain` during the bootstrap step itself. This directly overlaps with already-approved Decision K, whose entire point is that the executor — not Stalwart — generates the DKIM private key (via `openssl`, on host, under the executor's control), so that: (a) the plan retains full custody of the key file for the shred-after-apply cleanup discipline (Decision L), and (b) the DNS TXT record the plan publishes is guaranteed to be byte-for-byte the value Stalwart will actually sign with, captured from `stalwart-cli describe DkimSignature`'s server-derived `publicKey` field *after* the plan's own later `DkimSignature` upsert — not from whatever bootstrap-time auto-generation might produce, which the plan has no visibility into and no defined capture path for. If `generateDkimKeys: true` ran, the plan would either end up with an orphaned, unused, bootstrap-generated key sitting in Stalwart's config registry (dead weight, mild confusion for a future auditor) or — worse, unconfirmed — a naming/selector collision when the plan's own later `DkimSignature` upsert for selector `mail` runs. `false` avoids the ambiguity entirely: bootstrap completes with zero DKIM action taken, and Decision K's already-approved mechanism runs exactly as designed in the (corrected) Phase 3 proper.
- **`requestTlsCertificate: false`.** The live `Bootstrap` schema shows this also defaults to `true` ("Automatically Obtain TLS Certificate: Yes" in attempt 3's `get Bootstrap` output) with no documented detail on which ACME challenge type it uses. The most common default for an automatic first-run TLS flow is HTTP-01, which requires binding port 80 — already owned by nginx on this same host (serving `penpot.aiqadam.org` and `aiqadam.org` today). This is precisely the port-80-contention problem Decision F was designed to avoid, by deliberately deferring ACME configuration to a later, explicit Phase 4 step using DNS-01 against the Cloudflare token instead. Even if bootstrap's auto-request happens to also use DNS-01 or some other non-conflicting method, the plan has no confirmed knowledge of that, and letting an undocumented, automatic code path make a live ACME request (which is itself a rate-limited, externally-observable action against Let's Encrypt) is exactly the kind of "improvise past a genuine unknown" behavior this run's whole design lineage has been correcting against. `false` guarantees bootstrap completes cleanly, taking no TLS action, leaving 100% of TLS provisioning to the plan's own already-approved Phase 4.
- **Remaining `Bootstrap` fields** (`directory`, `dnsServer`, `dataStore`/`blobStore`/`inMemoryStore`/`searchStore`/`tracer`): kept at Stalwart's own shown live defaults (internal directory, manual DNS, RocksDB/local-disk storage backends) — this plan has no requirement for external directory (LDAP/SQL) or automated DNS-server integration, and changing these would be scope creep with no offsetting benefit. `secret` and `username` are server-set, not supplied.

This reasoning is judged correct and internally consistent with Decisions K and F as originally approved. **It is nonetheless flagged as requiring the user's own fresh sign-off, not silent auto-approval under the existing delegation** — see "Issues / risks" and "Open questions" below for why.

### Other corrections folded in (all empirically confirmed by attempt 3's executor, not guesses)

1. **Auth env var: `STALWART_PASSWORD`, not `STALWART_TOKEN`.** Live `--help` output confirms `--password` (Basic-auth, used with `--user admin`) reads from env var `STALWART_PASSWORD`; `STALWART_TOKEN` is for the separate, unused `--api-key`/Bearer-token auth mode. Corrected in Plan step 17 (Phase 3 apply) and the Phase 6 equivalent (step 34).
2. **`NetworkListener` protocol: `smtp` for port 587.** Live `describe NetworkListenerProtocol` confirms the enum is `smtp | lmtp | http | imap | pop3 | manageSieve` — no distinct "submission" variant. Port 587 (STARTTLS submission) uses `protocol: smtp`, `bind: "0.0.0.0:587"`, `useTls: true`, `tlsImplicit: false`. Corrected in Plan step 16's apply-plan construction.
3. **DKIM variant: `Dkim1Ed25519Sha256`, concretely, not "confirm live."** Live `describe DkimSignature` confirmed all four variants exist (`Dkim1Ed25519Sha256`, `Dkim1RsaSha256`, `Dkim2Ed25519Sha256`, `Dkim2RsaSha256`); both DKIM1 variants require `privateKey` (supplied) with server-derived `publicKey`, confirming Decision K's assumption. This plan now commits to `Dkim1Ed25519Sha256` specifically: DKIM1 (classic RFC 6376 selector-based signing) over DKIM2 because DKIM2 is described as "a newer/alternate signing scheme with an additional `flags` field" whose interoperability with mainstream receiving MTAs (Gmail, Outlook, etc. — the plan's actual deliverability targets) is unconfirmed and unnecessary for this task's requirements; Ed25519 over RSA because Ed25519 keys are shorter (simpler DNS TXT record, no multi-string splitting), current, and Stalwart's own DKIM1/Ed25519 pairing is a well-established, widely-interoperable combination (RFC 8463) — if a future receiving-side compatibility problem surfaces during Phase 8 deliverability testing, falling back to `Dkim1RsaSha256` is a same-shape, low-effort follow-on, not a re-architecture. No live "confirm and possibly switch" branch is needed anymore since the enum is now fully known.
4. **Safe-discovery technique for `certificateManagement`/`dkimManagement`/`dnsManagement`.** These three `Domain`-level inline object fields remain genuinely unconfirmed in shape — attempt 3's executor confirmed they are not independently `describe`-able by name in CLI 1.0.10 (`describe CertificateManagement` etc. all return "no object or enum named"). This plan does not guess their structure. Instead, Plan step 15a directs the executor to attempt a minimal `stalwart-cli create Domain --field name=aiqadam.org` call (omitting the three management fields entirely) first and treat the response as informative:
   - **If `create` succeeds outright** with the management fields left unset/defaulted, that is itself the answer (defaults are acceptable/inferred) — proceed, note the defaults observed via a follow-up `describe`/`get Domain` call.
   - **If `create` fails with a validation error naming the missing/malformed field(s) and describing their expected shape** (e.g., "field `dkimManagement` is required, expected one of: ...", or a JSON-shape hint) — this is useful live schema discovery, not a failure. The executor incorporates the disclosed shape into a corrected `create`/`apply` call and proceeds. Record the exact validation message in the execution log.
   - **If the response is a connection error, an auth error (401/403 unrelated to bootstrap-mode), or a server error (5xx)** — this is a genuine blocker. Halt and report; do not guess a payload shape to route around a server-side fault.
   - **If the response is the same "server is in bootstrap mode" rejection seen in attempt 3** — this indicates the new Bootstrap-completion step (Plan step 12 below) did not actually take effect; halt and report, do not retry blindly.
   This distinction (validation-error-as-discovery vs. hard-error-as-blocker) is stated explicitly here so the executor does not halt prematurely on a useful validation message, and does not improvise past a genuine unknown either.

- **MEDIUM-TO-HIGH — the `generateDkimKeys: false` / `requestTlsCertificate: false` resolution is a genuine, new judgment call, not a mechanism fix, even though it is well-reasoned and consistent with already-approved Decisions K and F.** This is the single item in this retry the orchestrator must not silently wave through under the user's attempt-3 delegation ("same-character mechanism fixes"). The distinction, stated plainly: a mechanism fix (auth env var name, protocol enum value, exact CLI flag spelling) changes *how* an already-agreed action is carried out and has no alternate reading of *what* happens. This item is different — it is the first concrete answer to a question the approved plan never actually posed before (because the `Bootstrap` object wasn't known to exist until attempt 3's executor found it): "what does bootstrap do with DKIM and TLS by default, and do we want that?" The plan's own answer (turn both off, defer entirely to already-approved later steps) is the reasoned, conservative choice and is very likely correct — but it is still a choice about server behavior with two boolean flags this repo has never previously configured, made by the solution-designer under a delegation that was granted for a different, narrower category of fix.
