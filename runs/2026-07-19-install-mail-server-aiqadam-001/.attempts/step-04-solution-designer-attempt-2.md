### New decisions this retry must make (corrected mechanics)

**A. Image: `stalwartlabs/stalwart:v0.16`, pinned.**
The archived `stalwartlabs/mail-server` repo is replaced by the actively maintained `stalwartlabs/stalwart` repo. Per this repo's own pinning convention (Penpot is pinned to `:2.16`, not `:latest`), and to avoid this exact failure class recurring on a future silent upstream change, this plan pins `v0.16` rather than `:latest`. If a newer minor is current at execution time, the executor should not silently substitute it — treat a version mismatch at execution time the same as attempt 1's image-not-found: halt and report, do not improvise (per "do not improvise" / defense-in-depth already established).

**B. Volumes: split `/etc/stalwart` (config) and `/var/lib/stalwart` (data), owned by UID 2000.**
Corrects attempt 1's single `/opt/stalwart:...` mount, which does not match this image's expected layout. The container runs as an unprivileged user (UID 2000), so host-side bind-mount directories must be `chown 2000:2000`, not `tvolodi:tvolodi` as attempt 1 did for `/opt/stalwart-mail`.

**C. Configuration: no hand-authored TOML — minimal `config.json` (datastore only) + non-interactive setup via `STALWART_RECOVERY_ADMIN`.**
v0.16 moved to a DB-backed config registry. Rather than relying on the interactive web setup wizard and scraping a randomly generated password from stderr logs (which an unattended executor cannot reliably do), this plan sets `STALWART_RECOVERY_ADMIN=admin:<password>` as a container environment variable, sourced from a new secret (`stalwart-mail-admin-password`, generated by the executor, stored per `secrets-inventory.md` convention, value never placed in this repo). This makes initial admin credentials deterministic and scriptable instead of dependent on log-scraping.

**D. DKIM key generation: via the web setup wizard's DKIM toggle, not a nonexistent CLI command.**
Attempt 1's `stalwart-cli dkim generate` command does not exist in this version. v0.16's setup wizard includes a toggle to auto-generate DKIM signing keys for the configured domain during initial setup. This plan drives that through the admin HTTP API after first boot (the setup wizard's actions are themselves exposed via the same admin API the web UI calls) rather than assuming a bare CLI binary exists in the container. The exact API call shape is confirmed at execution time against the running instance's own API — flagged as a normal execution-time detail (Issues/risks), not a design gap, matching the precedent already accepted for attempt 1's account-creation step.

**E. Ports: submission (587) explicitly enabled in addition to the v0.16 defaults (465/993).**
v0.16 does not enable plaintext-then-STARTTLS ports (587, 143) by default — only implicit-TLS 465/993. Since many mail clients still default to STARTTLS/587 for submission, this plan explicitly enables 587 during setup (via the admin API/UI listener configuration) in addition to accepting the 465/993 defaults. Port 25 (inbound SMTP, required for receiving mail from other mail servers regardless of client conventions) is also explicitly confirmed enabled — Stalwart enables 25 by default for MTA-to-MTA delivery, but this plan verifies it explicitly rather than assuming. IMAP over STARTTLS (143) and POP3 remain out of scope, consistent with the already-approved decision to skip POP3 and not expand scope.

**F. TLS: Stalwart's built-in ACME (internal), not externally-managed certbot mounted into the container.**
Decision: use Stalwart's internal ACME integration rather than continuing attempt 1's certbot-mount pattern. Reasoning:
- Attempt 1's failure mode was never actually TLS-related, but the certbot-mount pattern is a second class of path-mismatch bug (attempt 1's compose file mounted `/etc/letsencrypt/live/mail.aiqadam.org:/opt/stalwart/certs:ro` — a path inside the now-wrong `/opt/stalwart` volume root) that would need re-deriving correctly against the new `/etc/stalwart`+`/var/lib/stalwart` layout anyway.
- Internal ACME is Stalwart's documented, first-class TLS mechanism for this version and removes an entire class of mount-path bugs (the exact failure family that contributed to attempt 1's plan being harder to verify).
- Trade-off, stated plainly: this is a different TLS-management model than the certbot pattern used elsewhere on this host (Penpot, AiQadam prod, nginx vhosts all use certbot). This plan accepts that inconsistency for this one service in exchange for correctness and simplicity, rather than forcing Stalwart into a pattern its current version doesn't natively expect. If the user prefers operational consistency (one cert-issuance mechanism per host) over this plan's reasoning, that is a valid override to raise at approval — flagged in Open questions.
- Stalwart's internal ACME needs port 80 (HTTP-01) or DNS-01 reachable for issuance. Port 80 on `pro-data-tech-prod` is currently owned by nginx (serving the existing Penpot/AiQadam-prod HTTP-01 challenges and redirects). Rather than have two services race for port 80, this plan uses Stalwart's **DNS-01** challenge mode via the existing `cloudflare-ai-qadam-api-token` secret (already scoped to Zone.DNS edit on this exact zone) — Stalwart creates and cleans up its own `_acme-challenge.mail.aiqadam.org` TXT record automatically per renewal, with no nginx or port-80 contention. This is the same Cloudflare API token already in use for the DNS cutover phase, no new secret needed for this purpose.

**G. Disposition of the orphaned certbot cert from attempt 1.**
`/etc/letsencrypt/live/mail.aiqadam.org/` (ECDSA, valid until 2026-10-17, ~89 days as of the failed run) is left in place, untouched, as harmless dead weight — not reused, since this plan adopts internal ACME instead of the certbot-mount approach. It is not deleted either: deleting it would be a destructive action with no benefit (it doesn't conflict with anything, certbot's normal renewal cron will keep it valid or let it lapse harmlessly since no nginx vhost references it), and inventing a teardown step beyond what's needed matches the "don't improvise beyond the plan" discipline the executor already correctly applied in attempt 1. This is called out explicitly here rather than silently ignored, per the retry brief's requirement. Flagged in landscape update (step 08) as a known, harmless, orphaned artifact so a future host audit doesn't mistake it for live state.

**H. Admin UI path: verify at execution time, do not assume `/admin`.**
Attempt 1 assumed `https://mail.aiqadam.org/admin`. Current Stalwart docs are not conclusively confirmed on whether the webadmin is served at `/` or `/admin` for this version — this plan's nginx vhost (Phase 5) proxies the bare host (`https://mail.aiqadam.org/` → `127.0.0.1:8080/`) rather than a specific subpath, since proxying the root and letting Stalwart's own router handle its actual UI path is robust regardless of which path convention this version uses. The executor confirms the actual served path by hitting `http://127.0.0.1:8080/` directly on the host during Phase 1 bring-up (Plan step 7a) and records what it finds; this is a routine execution-time verification, not a blocker.
